Hi — we're going to walk the whole UniPilot system in five minutes, weighted toward the backend and how we test it. One sentence of framing: a Technion degree isn't a checklist you tick off, it's a rule system — buckets, pools, overlap rules, prerequisite chains. Almost everything hard about this project comes from taking those rules seriously. Let's start with why that's hard.

“What am I still missing to graduate?”

A question with no reliable answer today. Students reconcile a PDF transcript against catalog rules by hand.

Requirement buckets

Credits don't count once. Pool and overlap rules decide which bucket a course actually fills.

Prerequisite chains

Ordering constraints that span years, with equivalences across tracks.

Per-semester offerings

A course only exists if it's offered — winter, spring, or summer.

Schedule & exam conflicts

Lecture, tutorial and lab groups, plus exam dates that collide.

Get any one of these wrong and the answer is worse than useless — it's confidently wrong.

Here's the problem. Ask a Technion student what they're missing to graduate and there's no reliable answer. They're reconciling a PDF transcript against catalog rules by hand. Four things make it hard: credits don't count once — pool and overlap rules decide which bucket a course actually fills; prerequisite chains span years; a course only exists if it's offered that semester; and lecture, tutorial, lab and exam times all collide. And the key point — get any one wrong and the answer isn't just useless, it's confidently wrong. That's the constraint that shaped the entire backend.

The boundary is the design. Only web and api bind host ports. Everything else is service-to-service behind an X-Internal-Service-Token. Docker-first — the whole stack is docker compose up --build.

Now the backend. Eight services, but only two doors: web and api are the only things that bind a host port. Everything else — the agent, the transcript parser, the data-engineering CLI, inference, worker, the Outlook connector — is internal only, reachable service-to-service behind an internal service token. Mongo for persistence, Redis for rate limits. The boundary is deliberate: api is the only backend a client ever talks to, so there's exactly one place to enforce auth. And the whole stack comes up with a single docker compose command.

Here's the part people underestimate. We run on real Technion DDS catalog data, and it does not arrive clean. So there's a five-stage pipeline: export to a vault, import to staging, run quality checks, pass a promotion gate, then promote to production. That's about five thousand one hundred lines — genuinely more code to trust the data than to serve it. Promotion is guarded and reversible: dry-run first, and a parity check proves what shipped equals what was exported. This service is held at a hundred percent line coverage, no exceptions. A wrong catalog silently corrupts every answer downstream, so this is where the paranoia goes.

The API surface: seventy-five endpoints across auth, profile, catalog, transcript, progress, plans and risks. Everything except health is JWT-protected. Refresh tokens rotate and we detect reuse. Ownership is enforced per record — the user is the security boundary. Rate limiting is Redis-backed, internal services are gated by a service token, and the catalog is structurally read-only to students. And security isn't a claim here — there are ten dedicated security test suites that try to break exactly these boundaries.

Where the agent fits — and where it doesn't

The LLM may

  • Classify what you asked for
  • Extract preferences from your phrasing
  • Explain results the core computed
  • Propose an action

The LLM may not

  • Invent a requirement
  • Compute your progress
  • Decide what you've completed
  • Write student data — ever
agent proposesstudent confirmsapi executes the write

The agent is a view onto the deterministic core — it calls the same engines the REST endpoints do.

A word on the agent, because it's one part of the system, not the system. The language model may classify what you asked, extract preferences, explain results, and propose an action. It may not invent a requirement, compute your progress, decide what you've completed, or write student data — ever. Every action goes: agent proposes, student confirms, and the api executes the write. The agent is a view onto the deterministic core — it calls the exact same engines the REST endpoints do. Nothing it says is a new source of truth.

What UniPilot proves

Truth

Computed, never generated

Catalog rules are parsed into a typed DSL, and deterministic engines own every academic answer. The same inputs give the same result — explainable to a student and to a registrar.

Data

Trust is engineered

Real Technion data reaches a student only through a five-stage guarded pipeline with a promotion gate, a dry run, and a parity check. Roughly 5,100 lines of deliberate paranoia.

Proof

Tests are the warranty

5,273 backend tests, near 1:1 test-to-source, security and stress suites, and coverage gates that fail the build rather than warn.

One discipline runs through all three: the system only claims what it can compute.

To conclude — three things UniPilot proves. First, truth: academic answers are computed, never generated. Catalog rules become a typed DSL and deterministic engines own every answer, so the same inputs always give the same result, explainable to a student and to a registrar. Second, data: trust is engineered — real Technion data only reaches a student through a five-stage guarded pipeline with a gate, a dry run and a parity check. Third, proof: the tests are the warranty — over five thousand backend tests, near one-to-one test-to-source, and gates that fail the build rather than warn. One discipline runs through all three: the system only claims what it can compute. Thank you — happy to take questions.
